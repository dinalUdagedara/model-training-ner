Grey lines from Primary user are associations (who may use each use case) — not a reading order. Analytics and admin are separate bands; support is the bottom row.

If you feel lost

  • Start: left end of the top swimlane — Register / login (green “START” badge).
  • Order: follow blue dashed «include» arrows left → right in the User & documents row, then the same in Interview practice & feedback.
  • Primary user (grey lines): each line means that actor may participate in that use case — not a path to read in order. Sequence is only the blue «include» chains.
  • Analytics / admin bands: no horizontal include chain — separate goals or staff-only actions.

Layout notes

  • This figure uses horizontal swimlanes (rows) so document and practice flows read like a pipeline left to right.
  • Primary user has one association line per use case; crossing lines do not imply order.
  • Administrator links only to admin use cases.
  • Edit dashed AI / S3 lines if your thesis states NER is fully in-platform.
Figure notation — UML use case diagram (CrackInt)
Symbol / line UML meaning In this diagram
Large rectangle (thick border) System boundary — everything inside is the system under design. Labelled CrackInt System.
Rounded rectangles (grey / yellow / blue tint) Informal grouping (packages / swimlanes). Not standard UML; helps readability. User & documents, Interview practice, Analytics, Administration, Support — here as horizontal bands.
Stick figure Actor — human or external role interacting with the system. Job seeker, Student, Early career, Primary user, Career service professional, System administrator.
Rectangle with dashed border Often used for external system / secondary actor (not a person). AI service (e.g. OpenAI), Cloud storage (e.g. AWS S3).
Ellipse Use case — a goal the system supports for an actor. Named actions (e.g. Register / login, Parse résumé). Admin use cases use a darker stroke in this file.
Solid line, no arrowhead (actor ↔ use case) Association — “participates in” / “uses”. No direction required in UML for simple associations. Links from Primary user (one line per use case) to user-facing ovals; Career service to support/reports; Administrator to admin ovals.
Solid line + hollow triangle on the parent side Generalisation (between actors) — specialised role → more general role. From Job seeker, Student, Early career toward Primary user.
Blue dashed line + arrow + <<include>> (this file) Include — mandatory sub-steps: in this export read left → right in the documents and practice rows. Rows 1–2 only. Drawn on top of grey associations.
Grey solid lines (Primary user → each oval) Association — Primary user may participate in that use case. One line = one link; crossing lines do not mean sequence. Read order of work only from blue «include» (start at Register / login).
Dashed line + open arrow + stereotype <<include>> (standard UML) Same as include; colour is optional — some diagrams use black. In this HTML export, includes are blue for clarity.
«include» (guillemets) Stereotype — name of the relationship type in UML. Means “mandatory sub-behaviour” in the flow; not optional.
Not drawn here: <<extend>> Extend — optional extra behaviour under a condition (optional branch). Add only if you model optional paths (e.g. “get hint” only sometimes).
Dashed line (grey) to external box Common convention: dependency / communication with an external system (not a human actor). AI service and Cloud storage links to use cases that call APIs or store files. Your trained NER model is inside CrackInt — it is not this actor; it realises Parse résumé.
Support stub (grey) Same association; short vertical + branches to the two support ovals. From Primary user to Get help / Provide feedback.
